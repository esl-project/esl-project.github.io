Entries can be of different kinds depending on the content categories they contain. This one is a dummy example of a complete entry that contains information going from the functionality description down to the software implementation. The different main categories that can be contained in one entry are described in About ESL. Each entry may contain several categories, as in this example, but not more than one instance of each category. For example, It may contain an API, but could link to other APIs for the same functionality.

The box on the right is the one needed for a page containing a software implementation, which is both included (linked) and described in the entry. A simpler box is needed for non-software pages (see this example).

The entry is first introduced with a few lines describing its contents, i.e., an introductory abstract. It should be brief but sufficient to transmit to the reader what he/she needs to know in order to decide whether it is interesting for his/her project.

The table of contents appears automatically. In this example it is pushed as a float to the right. This is only optional. If not desired simply remove the three lines that follow this sentence in the page source.

Functionality / Theory / Method

Here the basic functionality, methodology or theory behind the software is described. If present, the entry should include Functionalities in the list of categories at the end of the page. If the entry contained, e.g., only the description of an API, it should point to the entry in which the functionality is described. Depending on the kind of entry this section can vary in length and even be absent if the function of the software is standard or obvious from the initial description above. Algorithm

If the method described is quite generic and allows for different ways of its being implemented, a description of (or link to an entry describing) the algorithm is important and useful. In some cases it is obvious and not needed. If present, include Algorithms in the list of categories at the end of the page. It can be described with words, flow charts or metalanguage.

Programming interface

If present, APIs should be included in the list of categories at the end of the page. Alternatively, a link to another entry with the API description can be put here.

There is also the category Generic interfaces, which would correspond to a section with the same name, but describing the interface in less detail/abstract terms.

subroutine theRoutine( a, B, result, n, w)

Implementation independent:

  • a (input) INTEGER
    Variable .
  • B (input) DOUBLE PRECISION array, dimension (a,a)
    Matrix .
  • result (output) DOUBLE PRECISION array, dimension (a,a)
    The result.

Algorithm specific:

  • n (output) DOUBLE PRECISION
    Variable .

Implementation specific:

  • w (input, optional) LOGICAL
    T: Do this.
    F: Don't do it.
