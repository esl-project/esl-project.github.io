ESL — The Electronic Structure Library is a community-maintained repository of software of use for electronic structure simulations, constituting an extended library that can be employed by everyone for building their own packages and projects. It consists of entries documenting methods, functionalities, interfaces, standards and pieces of code ranging from small routines for performing simple tasks, all the way up to complete libraries.

The ambition of the ESL is to segregate layers of functionality within modules which are general, standardised and efficient. In this way, new ideas, and new science, can be coded by scientists without needing to rewrite functionalities that are already well-established, and without needing to know more software engineering than science. In other words, we want to separate the coding effort for cutting-edge research from the software infrastructure it rests on top of, which needs maintaining and rewriting at every step of the hardware race.

Contents of ESL

ESL is not a conventional computer library. Allowing it to grow with contributions open to a wide community prevents imposing the rigidity implied by the a priori definition of a conventional library. It is neither a software repository, however, or at least it is not intended to be that. The expectation is for ESL to be useful and used, and through that slowly to standardise software, APIs, data, of common use in the community, as opposed to accumulate random bits of code.

ESL contents are categorised as Functionalities, Algorithms, Generic interfaces, APIs, Data standards, and Software (other categories may be added if needed). In addition, several instances of the same category can be joined in a bundle. They are all defined as follows.

Functionalities

This describe the particular functionality of the related software. Depending on what it is, it describes the theory behind, a method etc.

Algorithms

Describes an algorithm in detail, including the possibility (desirable) of describing it with meta-language.

Generic interfaces

Describes the generic interface for the possible software associated with a given functionality. Concise, human-readable, it does not need to prescribe details as data types etc.

APIs

APIs (application programming interfaces) describes the above with all level of detail that is necessary for some other software to link with the one described by the API.

Data standards

Contains the specification of how different kinds of data are stored in files, containing all necessary details for any program to be able to read and write according to the standard.

Software

Actual piece of software implementing the above. Here the software is documented in all the details not covered by the previous categories.

ESL entries

There is flexibility on what an entry can contain. In one extreme one entry would include one instance of any of the categories above (e.g. a detailed description of an API), then linking to another entry describing the functionality or the algorithm, and another for the implementation. This kind of separation promotes the possibility of e.g. more than one implementation for the same API (as happens, for instance with the several implementations of LAPACK or SCALAPACK by Intel or IBM etc).

One entry can also, however, contain instances of several categories at once. For example, if a completed piece of software is documented, the entry can contain the functionality, the algorithm, the API and the software itself.

If, in an independent posterior development, a different implementation is produced for the same functionality and API, a link to a new entry can be introduced in the appropriate section of the original page, pointing to the new implementation, and a new entry is introduced with the software implementation (and its description) only.

Category tags

It is important that any entry is tagged with the categories of its contents. In this example of an entry just describing an API, the page source should contain [[Category:APIs]] at the end. If the entry contains information corresponding to different categories, all of them should be similarly tagged (e.g. [[Category: Functionalities]] [[Category:Algorithms]] [[Category:APIs]] [[Category:Software]] in this example).

Category sections

Contents of different categories should be clearly defined in the corresponding sections of the entries, as illustrated in the dummy and real examples.

Basic rule: uniqueness

Although one entry may contain descriptions for several categories, it is important that either none or only one instance is described for each category. Other instances should be in separate pages with appropriate links.

Bundles

Entries categorised as bundles can contain collections of any of the categories, as for example, in a multipurpose (sub)library, or a list of available eigensolvers. Ideally bundles should mainly list the contents they bundle with links to theor separate description, be it at whichever of the possible category levels.
